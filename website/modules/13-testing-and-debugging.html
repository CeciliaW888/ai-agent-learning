Day 13: Testing & Debugging Agents

🐛
Agents are harder to debug than regular code. They're non-deterministic — the same input can produce different tool calls, different reasoning paths, and different outputs each time. You can't just set a breakpoint. You need to trace the agent's thinking step by step.

🎯 Learning Objectives

1
Read an agent reasoning trace

Enable verbose=True and understand what Thought → Action → Observation means at each step.

2
Add structured logging

Log every tool call and result so you can audit what your agent did — even after the fact.

3
Handle tool failures gracefully

Wrap tool calls with try/except and return clear error messages — so the agent can reason about failures instead of crashing.

📚 Core Concepts

🔍 Tracing Agent Reasoning with verbose=True

The fastest way to start debugging is to turn on verbose output. This prints the full Thought → Action → Observation chain so you can see exactly what your agent is thinking at every step.

🚨 The 5 Most Common Agent Bugs

1
Infinite Loop

Agent keeps calling tools without ever giving a final answer. Fix: set recursion_limit=25 in your config and add clear stopping criteria in the system prompt.

2
Wrong Tool Selection

Agent searches when it should calculate, or vice versa. Fix: make tool descriptions mutually exclusive — explicitly say what each tool should NOT be used for.

3
Hallucinated Tool Calls

Agent tries to call a tool that doesn't exist, or passes wrong argument types. Fix: validate tool calls before execution and return a helpful error message.

4
Context Window Overflow

Agent starts forgetting earlier context as the conversation grows. Fix: monitor token count and summarize older messages before they overflow.

5
Silent Tool Failures

A tool quietly returns bad data, and the agent produces wrong results. Fix: always validate tool outputs and return explicit error strings the agent can reason about.

⚠️
"An agent that fails silently is worse than one that fails loudly." When a tool crashes with a Python exception, at least you know something went wrong. When it returns garbage data without any indication of failure, the agent confidently produces wrong answers — and you have no idea why.

🛡️ Error Handling Pattern

Here's the standard error handling pattern that keeps your tools from failing silently. Notice how errors are returned as strings — so the LLM can read them and try something different:

import logging

logger = logging.getLogger("research_agent")

@tool
def search_web(query: str) -> str:
    """Search the web for current information.
    Returns top 3 results with titles and summaries.
    """
    logger.info(f"SEARCH called: '{query}'")

    try:
        from tavily import TavilyClient
        client = TavilyClient(api_key=os.getenv("TAVILY_API_KEY"))

        if not query or len(query.strip()) < 3:
            return "Error: Query too short. Please provide a more specific search term."

        results = client.search(query, max_results=3)

        if not results.get("results"):
            logger.warning(f"No results for: '{query}'")
            return f"No results found for '{query}'. Try a more specific query."

        formatted = []
        for r in results["results"]:
            formatted.append(f"**{r['title']}**\n{r['content'][:300]}\nSource: {r['url']}")

        result_str = "\n\n---\n\n".join(formatted)
        logger.info(f"SEARCH returned {len(result_str)} chars")
        return result_str

    except requests.Timeout:
        logger.error("Search timed out")
        return "Error: Search timed out. Try again or use a simpler query."
    except Exception as e:
        logger.error(f"Search failed: {e}")
        return f"Error: Search failed — {str(e)}. Try rephrasing the query."

💬 A Debugging Session in Action

Here's what it looks like when a developer reads an agent trace to diagnose a failure and fix it:

Dev runs: agent.invoke({"messages": [{"role": "user", "content": ""}]})
Agent [Thought]: The user wants... hmm, the query is empty. I'll search for something anyway.
Agent [Action]: search_web("") → Tool crashes with KeyError: no results for empty string
Dev reads trace: "Ah — the tool doesn't validate empty input. The agent can't reason about the failure because it got an exception, not an error string."
Dev fixes tool: adds `if not query: return "Error: Query cannot be empty. Please provide a search term."` at the top of search_web.
Agent [Thought after fix]: The search returned "Error: Query cannot be empty." — the user's message must have been blank. I should ask for clarification.
Agent: It looks like your message came through empty. What topic would you like me to research?
Dev: Much better! The agent now handles the edge case gracefully instead of crashing.

🔄 No Error Handling vs Graceful Fallbacks

💥 Agent with No Error Handling

Tool throws a Python exception → agent crashes → user sees a stack trace or a hung process. No information about what went wrong or how to recover.

Tool fails silently or crashes → agent spirals or loops → you have no idea why.

Debugging this is painful because you can't tell where the failure originated.

🛡️ Agent with Graceful Fallbacks

Tool catches errors and returns a descriptive string → agent reads the error → agent adapts its strategy (tries a different query, asks for clarification, or explains the problem).

"Error: No results for 'xyzzy'. Try rephrasing." → Agent: "I couldn't find information on that — could you clarify what you're looking for?"

Logging makes every step auditable after the fact.

📖 Key Terms

Verbose Mode
Setting verbose=True on an agent executor to print the full Thought → Action → Observation chain — the fastest way to see what's going wrong.
LangSmith
LangChain's full observability dashboard — shows every LLM call, tool call, token count, and error across all runs. Free for development.
Non-deterministic
Same input may produce different outputs each run. This is why agent testing is harder than regular unit testing.
Recursion Limit
The maximum number of agent steps allowed before the system stops. Prevents infinite loops. Set in LangGraph's config.
Edge Case
An unusual input or situation the agent wasn't designed for — empty queries, extremely long messages, nonexistent topics. These are where bugs hide.
Audit Trail
A log of every decision and tool call the agent made during a run. Essential for debugging and for building trust in agent systems.
