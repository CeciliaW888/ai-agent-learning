Day 5: Memory & Context

🧠
The big idea: Without memory, every conversation with an agent starts from zero. It doesn't know your name, your preferences, or what you worked on yesterday. Memory is what turns a clever tool into a genuine assistant.

🎯 Learning Objectives

1
Understand the three types of agent memory

Short-term, long-term, and working memory each solve a different problem — and using the wrong one leads to bugs.

2
Know how vector databases enable long-term memory

Semantic search lets agents find relevant memories by meaning, not just exact keywords.

3
Recognize the difference between stateless and stateful agents

Stateful agents learn and improve across sessions. Stateless agents are always strangers.

📚 Core Concepts

🗄️ The Three Types of Agent Memory

Think about how you remember things. Right now you're holding this lesson in active focus (short-term). You also know your own name and how to ride a bike — that's long-term. And if you're solving a math problem you might jot intermediate steps on paper — that's working memory. Agents need all three.

💬
Short-Term Memory — Conversation Buffer

The current conversation. Every message exchanged this session is appended to a list and included in the next prompt. Simple and fast — but has a hard ceiling (the context window). Long conversations must be summarized or truncated.

💾
Long-Term Memory — Vector DB or Files

Information that persists between sessions — user preferences, learned facts, past research. Retrieved by semantic search (meaning, not exact match), so the agent can find "Python memories" even when you ask about "preferred programming language."

📋
Working Memory — Task Scratchpad

Temporary storage for the current task — intermediate search results, a running plan, half-finished sections. Erased when the task is done, like a whiteboard you clear after a meeting.

💡
Why can't we just give the LLM everything at once? LLMs have a context window limit — a maximum amount of text they can process. GPT-4 handles about 128K tokens, which sounds like a lot until you have months of conversation history, tool results, a long system prompt, and the current question all competing for space. Memory systems solve this by choosing what to include.

🔍 How Long-Term Memory Works (Vector Databases)

Vector databases store memories not as plain text but as embeddings — lists of numbers that capture semantic meaning. When you search, your query is also converted to numbers, and the database finds the most mathematically similar stored memories. This is why you can ask "what programming language does the user prefer?" and retrieve a memory that says "user likes Python for backend" — even though the words are completely different.

1
Store a memory

Text like "User prefers Python for backend development" is converted to a vector (e.g. [0.23, -0.45, 0.87, ...]) and saved alongside the original text.

2
Search by meaning

Your query becomes a vector. The database returns the stored memories whose vectors are closest — the most semantically similar ones, regardless of exact wording.

3
Inject into the prompt

Retrieved memories are added to the system prompt before the LLM sees the new message: "Here's what I know about this user..." — context without repetition.

💬 Watching Memory in Action

Here's what it looks like when an agent uses long-term memory to recognize a returning user and pick up right where they left off:

User: Hey, I need help with my project again.
Agent [Recalling memories...]: User prefers Python. Last session: building a web scraper. Goal: data pipeline for their startup. Prefers concise answers.
Agent: Welcome back! Last time we were building your Python web scraper for the startup data pipeline. Ready to continue, or something new?
User: I finished the scraper! Now I need to store the results in a database.
Agent [Thought]: They finished the scraper — great progress. I know they use Python and are building for a startup, so I should suggest something practical and lightweight to start.
Agent: Perfect next step. Since you're on Python and building a startup pipeline, I'd start with SQLite for local development and PostgreSQL when you're ready to scale. Want a schema that fits the kind of data your scraper collects?
User: That's exactly what I needed. You remembered everything from last time!
Agent: That's long-term memory at work — I retrieved your preferences and project context from our previous session so you didn't have to re-explain. [REMEMBER: User completed the Python web scraper. Now building database storage layer with SQLite → PostgreSQL path.]

Notice: The agent didn't ask "who are you?" or "what are you building?" — long-term memory bridged the gap between sessions, making it feel like a real colleague rather than a stranger every time.

📬 Stateless vs Stateful

📭 Stateless Agent (No Memory)

Starts completely fresh with every conversation. No idea who you are, what you've worked on, or what you prefer. Every session is a first meeting — forever.

You: "Help me with my Python project."

Agent: "Sure! What's your project about? What language are you using?" — every single time.

⚠️
Memory pitfalls to watch out for: Storing too much slows retrieval and confuses the agent with noise. Stale memories ("user lives in Melbourne") can mislead — add timestamps and expiry. Conflicting memories ("user likes Python" vs "user switched to Rust") need a recency strategy. And long-term memory means storing personal data — be thoughtful about privacy and what you actually need to retain.

📖 Key Terms

Short-Term Memory
The current conversation history, included in every prompt. Resets when the session ends.
Long-Term Memory
Persistent storage across sessions — vector DB, files, or key-value stores. Retrieved by semantic search when relevant.
Working Memory
Temporary scratchpad for the current task. Intermediate results, running plans, notes. Cleared when the task is done.
Vector Database
A database that stores and retrieves text by semantic similarity — finding memories by meaning, not exact keyword match. Examples: Chroma, Pinecone, Qdrant.
Embedding
A numerical vector representation of text that captures its semantic meaning. The foundation of how long-term memory retrieval works.
Context Window
The maximum amount of text an LLM can process at once. Memory systems choose carefully what fits inside this limit.

🤔 Reflection

Design a memory system for a personal AI assistant. What should go in short-term memory? What in long-term? What should it never store? Think through the privacy implications — if the agent remembers everything, what could go wrong?
