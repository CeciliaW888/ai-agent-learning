📚 Core Concepts

🧩 Why LangChain Exists

You could call the OpenAI API directly with raw Python — but you'd end up rebuilding the same plumbing in every project: prompt management, tool execution, conversation history, output parsing. LangChain handles all of that so you can focus on what your agent does, not how it talks to an API.

Think of it this way: you could build a website with raw HTTP sockets, but you'd use React or Django instead. LangChain is the React of AI agents.

💡
The LangChain ecosystem at a glance: langchain-core (base abstractions) → langchain (chains and agents) → langchain-openai / langchain-anthropic (model providers) → langgraph (graph-based workflows) → langsmith (debugging and evaluation). You don't need all of these on day one.

🔗 Building Block: Chains

A Chain connects components like Unix pipes. The | operator does the work — each piece feeds its output into the next:

from langchain_openai import ChatOpenAI
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import StrOutputParser

llm = ChatOpenAI(model="gpt-4o-mini", temperature=0)

prompt = ChatPromptTemplate.from_messages([
    ("system", "You explain {topic} in simple, clear terms."),
    ("user", "{question}")
])

# The | operator creates a chain: prompt → LLM → parser
chain = prompt | llm | StrOutputParser()

result = chain.invoke({
    "topic": "machine learning",
    "question": "What is overfitting?"
})
print(result)  # clean string output, no extra formatting needed

🤖 Building Block: Agents

An Agent is a chain that can loop. You give it tools, and the LLM decides what to call and when. create_react_agent from LangGraph gives you the full PDA loop in a single call:

from langchain_openai import ChatOpenAI
from langchain_core.tools import tool
from langgraph.prebuilt import create_react_agent

llm = ChatOpenAI(model="gpt-4o-mini", temperature=0)

@tool
def search(query: str) -> str:
    """Search the web for information about any topic."""
    return f"[Simulated results for '{query}']"

@tool
def calculate(expression: str) -> str:
    """Calculate a math expression like '100 * 0.15' or '(50 + 30) / 2'."""
    import ast
    try:
        return str(ast.literal_eval(expression))
    except Exception as e:
        return f"Error: {e}"

agent = create_react_agent(llm, [search, calculate])

result = agent.invoke({
    "messages": [{"role": "user", "content": "What's 15% tip on a $85 dinner?"}]
})

for message in result["messages"]:
    print(f"{message.type}: {message.content}")
💡
Notice the docstrings on each tool. LangChain passes them directly to the LLM as tool descriptions. The LLM decides which tool to use based on these descriptions — a vague docstring means an agent that picks the wrong tool or calls tools at random.

💬 Watching a Chain Execute Step by Step

Here's what happens inside LangChain when you invoke an agent on a real request:

User: "What's 20% of $340, and search for current interest rates."
LangChain: Formatting tool schemas into JSON and sending everything to the LLM...
LLM [Thought]: The user wants two things — a calculation and a search. I'll handle the math first.
LLM [Action]: calculate("340 * 0.20")
Tool Result: "340 * 0.20 = 68.0"
LLM [Thought]: Got the calculation. Now I need to search for interest rates.
LLM [Action]: search("current interest rates 2024")
Tool Result: "[Search results: Fed funds rate is X%, mortgage rates are Y%...]"
LLM [Thought]: I have both pieces of information now. I can give a complete final answer.
Agent [Final Answer]: 20% of $340 is $68. As for interest rates, the Fed funds rate is currently... [summarizes search results]

The key insight: LangChain handles the loop automatically. You define the tools — it handles the back-and-forth between the LLM and those tools until the task is done.

🔄 Chain vs Agent vs Graph

🔗
Chain — Fixed pipeline

Always does A → B → C. High predictability, zero flexibility. Best for: document summarization, formatting pipelines, fixed Q&A workflows.

🤖
Agent — LLM decides the path

The LLM picks which tools to call and when. Medium predictability, high flexibility. Best for: open-ended research, multi-step reasoning, tasks where you don't know the steps in advance.

🗺️
Graph (LangGraph) — Developer designs the flow

You define exact nodes and edges. High predictability, full control. Best for: complex production workflows with approvals, branching, and human-in-the-loop checkpoints. Week 3 topic.

🃏 LangChain Chain vs LangChain Agent

🔗 LangChain Chain

A fixed sequence of steps connected with |. Every invocation follows the same path: prompt → LLM → output parser.

Analogy: An assembly line — the product always goes through the same stations in the same order.

Use when: You know exactly what steps are needed and they don't change based on the input.

🤖 LangChain Agent

A loop where the LLM decides what to do next. It picks tools, calls them, reads results, and decides whether to keep going or give a final answer.

Analogy: A freelancer — given a goal, they figure out the steps and adapt as they go.

Use when: The steps needed depend on the specific task or on the results of earlier steps.

📖 Key Terms

LangChain
Python/JS framework that handles the plumbing of LLM apps — prompts, tools, memory, agent loops — so you focus on logic, not infrastructure.
Chain
A linear sequence of components connected with |. Fixed flow every time: prompt → LLM → parser. Predictable, simple, no loops.
Agent
An LLM in a decision loop. It decides which tools to call, observes results, and loops until done. Built on the ReAct pattern from Week 1.
Tool
A Python function decorated with @tool that the agent can call. The docstring becomes the description the LLM uses to decide when to call it.
Prompt Template
A reusable prompt with {variable} placeholders. Fill in different values each call — same structure, different content every time.
LangGraph
An extension of LangChain for graph-based workflows. You design nodes and edges — maximum control for complex production agents. Week 3 topic.
