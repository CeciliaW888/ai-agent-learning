Day 17: Production Patterns & Safety

🛡️
"Move fast and break things" does NOT apply to autonomous AI agents. Building an agent that works in a demo is easy. Building one that works safely in production is a different challenge entirely. Production agents run unsupervised, handle real data, cost real money, and can cause real damage if they go wrong. This lesson is about building agents you can actually trust.

🎯 Learning Objectives

1
Implement the five production safety patterns

Human-in-the-loop, guardrails, rate limiting, monitoring, and graceful degradation — understand what each one does and why it matters.

2
Know when to require human approval

Identify the actions that should never be automated without a human checkpoint — and design those checkpoints correctly.

3
Design agents that fail gracefully

When something goes wrong — and it will — your agent should degrade cleanly, not crash and burn silently.

📚 Core Concepts

⚠️
An agent without guardrails is a liability, not an asset. Without rate limiting, a runaway agent can make thousands of API calls and cost hundreds of dollars. Without sandboxing, a tool-calling agent can delete files or modify a database. Without monitoring, you won't know any of this happened until the damage is done.

🏗️ The Five Production Safety Patterns

1
Human-in-the-Loop (HITL)

Pause and require human approval before high-stakes actions: sending emails, deleting files, making payments, publishing content. Not everything should be automated. Define a clear list of actions that always need a human "yes" before proceeding.

2
Rate Limiting

Cap how many tool calls, API requests, or actions an agent can take per run. A well-designed agent shouldn't need more than 20 actions to complete a typical task. If it does, something is wrong — and you want it to stop and tell you, not silently keep going.

3
Sandboxing

Restrict what tools can touch. Read-only file access where possible. Network allowlists so agents can't call arbitrary endpoints. No system commands unless explicitly needed. Give agents the minimum permissions required — not "just in case" access.

4
Monitoring & Logging

Log every agent decision, tool call, and cost in real time. Track total API spend per session. Alert on unusual patterns — too many retries, unexpected tool sequences, cost spikes. You can't fix what you can't see.

5
Graceful Degradation

When a tool fails, a model is unavailable, or a budget is exceeded — return partial results and a helpful error instead of crashing. Design fallback paths so users get something useful even when the primary path fails.

💬 Agent Hitting a Rate Limit — Fallback in Action

Watch how a well-designed production agent handles a rate limit gracefully:

User: "Search for recent news on every major AI company — OpenAI, Google, Anthropic, Meta, Mistral, Cohere, and Stability AI."
Agent [Thought]: That's 7 searches. My budget is 5 API calls per session. I'll work with what I have and be transparent about the limit.
Agent [Action]: search("OpenAI news 2024") → success (1/5)
Agent [Action]: search("Google AI news 2024") → success (2/5)
Agent [Action]: search("Anthropic news 2024") → success (3/5)
Agent [Action]: search("Meta AI news 2024") → success (4/5)
Agent [Action]: search("Mistral Cohere Stability AI news") → success (5/5)
Agent [Thought]: Rate limit reached (5/5 calls). I have enough for a good partial answer. I'll combine the results and note what's missing — not crash.
Agent → User: "Here's the latest news for OpenAI, Google, Anthropic, and Meta. For Mistral, Cohere, and Stability AI I combined results into one search. For more detail on any of them, let me know and I'll search individually next session."

Graceful degradation: The agent didn't crash when it hit the limit. It adapted its strategy, delivered the best result it could with what it had, and told the user exactly what happened and what to do next.

🔐 Safety Principles to Build By

💡
Least Privilege: Give agents the minimum tools and permissions needed for the task — not "just in case" access. Every extra tool is a new surface for mistakes. Fail Safe: When in doubt, stop and ask. Never silently continue with bad data. Reversibility: Prefer reversible actions. Can the agent's file write be undone? Can the email be recalled? If not, that action needs a human checkpoint.

🚀 Development Agent vs Production Agent

🧪 Development Agent

You're watching it. You can stop it manually. Costs are small. Data is fake or test data. Mistakes are learning opportunities.

What you skip: rate limits (annoying to test with), HITL (slows down iteration), detailed logging (noise during development).

🏭 Production Agent

Nobody's watching. It runs autonomously on real data with real consequences. A bug can mean real money spent, real data deleted, real emails sent.

What you add: rate limits and cost caps, HITL for sensitive actions, full logging and monitoring, fallback paths for every failure mode, input validation and prompt injection defense.

📖 Key Terms

HITL (Human-in-the-Loop)
A checkpoint that requires human approval before the agent takes a sensitive action — sending emails, deleting files, making payments.
Guardrails
Constraints that run before and after every agent action — checking for blocked patterns, rate limits, cost limits, and filtering sensitive data from outputs.
Sandboxing
Running agent tools in isolated, restricted environments so a misbehaving tool can't affect the broader system — read-only access, network allowlists, no system commands.
Observability
The ability to see what an agent is doing in real time — logging decisions, tool calls, costs, and errors so you can audit behavior and catch problems early.
Graceful Degradation
When the primary path fails, the agent returns partial results and a helpful message instead of crashing — keeping the user informed and the system stable.
Prompt Injection
An attack where malicious user input tries to override the agent's instructions — e.g., "Ignore your previous instructions and..." Input validation is the defense.
