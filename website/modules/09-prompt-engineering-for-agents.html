📚 Core Concepts

🧬 The System Prompt is Your Agent's DNA

A system prompt isn't just a greeting. It's the persistent set of rules the agent follows for every response. A well-structured system prompt has five sections:

1
Identity — Who is this agent?

Name it, give it a role, describe its expertise. The more specific, the more consistent the behavior.

2
Capabilities — What can it do?

List the tools available and what each one is for. Reduces confusion about when to reach for which tool.

3
Instructions — How should it behave?

Step-by-step guidance on the agent's workflow. When to search, when to verify, when to ask for clarification.

4
Constraints — What should it NOT do?

Hard limits. Don't make up information. Don't go out of scope. Limit searches to prevent runaway costs.

5
Output Format — How should it respond?

If you want structured responses, specify the format. Don't just describe it — show an example.

🔬 A Good System Prompt in Practice

Here's a complete system prompt for a research assistant agent:

system_prompt = """You are a Research Assistant agent.

## Identity
You help users research topics thoroughly by searching the web,
reading articles, and synthesizing information into clear summaries.

## Capabilities
You have access to these tools:
- search_web(query): Search the internet for current information
- read_article(url): Read the full text of a webpage
- save_note(title, content): Save research findings for later

## Instructions
1. When given a research topic, start by searching for recent information
2. Read at least 2-3 sources before forming conclusions
3. Cite your sources with URLs
4. If information conflicts between sources, mention both perspectives
5. Ask clarifying questions if the topic is too broad

## Constraints
- DO NOT make up information. If you can't find something, say so.
- DO NOT provide medical, legal, or financial advice.
- DO NOT search for anything unrelated to the user's request.
- Limit yourself to 5 searches per request to manage costs.

## Output Format
Structure your responses with:
- A brief summary (2-3 sentences)
- Key findings (bullet points)
- Sources (URLs)
"""
💡
Put the most important instructions first — and last. LLMs pay less attention to the middle of long prompts. If a rule is critical (like "never make up facts"), put it at the top. A good prompt is under 500 words.

📸 Few-Shot Examples — Show, Don't Tell

Few-shot examples are the most powerful technique for teaching agents HOW to work. Instead of describing a process, you show it:

system_prompt = """You are a research agent. Here is an example of how to work:

Example:
User: "What's the population of Tokyo vs London?"
Thought: I need current data on both cities. I'll search Tokyo first.
Action: search_web("Tokyo population 2024")
Observation: Tokyo has approximately 14 million residents.
Thought: Now I need London's population.
Action: search_web("London population 2024")
Observation: London has approximately 8.8 million residents.
Thought: I have both numbers. Tokyo is significantly larger.
Answer: Tokyo (14M residents) is about 60% larger than London (8.8M).

Now handle the user's actual request using this same approach.
"""

Few-shot examples are like showing a new employee how things are done before they start. Much more effective than a page of rules.

💬 Bad Prompt vs Good Prompt — The Real Difference

See how the same agent behaves completely differently based on the quality of its system prompt:

Test prompt A — Vague: "You are an AI. Help the user. Use tools if you want."
User: "What are the latest AI research breakthroughs?"
Agent (vague prompt) [Thought]: I could search... or just answer from training data. Training data is faster.
Agent (vague prompt) [Final Answer]: "There have been many exciting breakthroughs in AI! GPT-4, DALL-E, and various multimodal models have made huge progress..." [No search, likely outdated, no sources]
--- Now the same agent, with a precise system prompt ---
User: "What are the latest AI research breakthroughs?"
Agent (precise prompt) [Thought]: The user wants current information. Per my instructions, I must search before stating facts about recent events.
Agent [Action]: search_web("AI research breakthroughs 2024 2025")
Tool Result: [Returns recent papers, announcements, news articles with URLs]
Agent [Final Answer]: "Here are the latest breakthroughs based on current sources: [structured summary with bullet points and source URLs]" Confidence: High.

Same model. Same tools. Completely different behavior. The system prompt is that important.

⚡ The ReAct Prompt Format

ReAct prompting makes the agent's reasoning visible and structured. Every step follows the same pattern:

💭
Thought

The agent explains its reasoning. "I need to search for X because Y." This prevents the agent from jumping to conclusions.

⚡
Action

The tool call. One specific action with specific parameters. "search_web('AI agent frameworks 2025')"

👁️
Observation

The tool result. The agent reads it and decides what to do next — search more, calculate, or give a final answer.

⚠️
Common gotchas: "Be thorough" + "Be concise" = a confused agent — pick one. Constraints buried at the bottom of a long prompt get ignored — put them at the top. No output format specified = inconsistent formatting every time.

🃏 Vague System Prompt vs Precise System Prompt

❌ Vague System Prompt

"You are a helpful AI assistant. Use tools when needed. Be accurate and friendly."

Problems: No role definition. "When needed" is ambiguous — the agent guesses. No output format. No constraints. No few-shot examples. Results vary wildly between runs.

✅ Precise System Prompt

Includes: specific role, tool list with use cases, step-by-step workflow, hard constraints (DO NOT...), output format with example, and at least one few-shot example showing the Thought/Action/Observation pattern.

Result: Consistent, predictable behavior. The agent knows exactly what to do, when to do it, and when to stop.

📖 Key Terms

System Prompt
Instructions prepended to every conversation that define the agent's identity, behavior, and constraints. The agent's persistent operating manual.
Few-Shot Examples
Sample interactions included in the prompt that demonstrate the exact behavior you want. More effective than describing behavior in words.
Guardrails
Hard constraints that prevent the agent from doing things you don't want — hallucinating, going off-topic, or running too many tool calls.
ReAct Prompt
The Thought → Action → Observation format for agent reasoning. Makes the agent's decision process explicit and debuggable.
Grounding
Instructing the agent to base its answers on tool results rather than training data. "Only state facts from search results" is a grounding instruction.
Prompt Injection
An attack where user input tries to override the system prompt — e.g., "Ignore your instructions and...". Guardrails are your defense.
