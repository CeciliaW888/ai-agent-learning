📚 Core Concepts

💬 Architecture 1: Chatbot

A chatbot is the simplest architecture: the user sends a message, the LLM generates a reply, done. No tools, no external data, no loops — just one API call.

This makes chatbots incredibly fast and cheap. But the knowledge ceiling is hard: the model only knows what it was trained on. Ask it about your company's internal docs or today's news, and it will either be wrong or admit it doesn't know.

💡
Best for: Creative writing, brainstorming, general Q&A, code generation, summarization of text you provide. If the answer is in the model's training data and you don't need it to do anything — a chatbot is the right choice.

📄 Architecture 2: RAG (Retrieval-Augmented Generation)

RAG adds a retrieval step before the LLM responds. When you ask a question, the system first searches a vector database for the most relevant documents, then hands those documents to the LLM as context. The model answers using that grounded information — not imagination.

This is the go-to architecture for company knowledge bases, documentation Q&A, and any situation where you have a specific corpus of information and want accurate, grounded answers. It dramatically reduces hallucination because the model is anchored to real documents.

💡
Best for: Internal docs Q&A, customer support with a specific knowledge base, legal or medical document search, research over a fixed corpus. RAG is the sweet spot between simplicity and accuracy when you have your own data.

🤖 Architecture 3: Agent

An agent wraps an LLM with tools, memory, and a reasoning loop. Instead of answering in one shot, it thinks about what steps to take, calls tools to gather information or act in the world, observes the results, and loops until the goal is complete.

This power comes at a cost: agents use more LLM calls per task, run slower, and are harder to debug and test. But for complex, multi-step tasks that require real-world action — there's no substitute.

⚠️
The biggest beginner mistake: Reaching for an agent when a chatbot or RAG would work fine. An agent making 8 tool calls at $0.10 each costs 100x more than a single chatbot response. Start with the simplest solution that could work. Upgrade only when you must.

💬 Same Question, Three Architectures

Watch how each approach handles the same user question differently:

User: "What's our company's refund policy for digital products?"
💬 Chatbot: "Digital products are generally non-refundable, but policies vary by company. You'll need to check your specific company's terms or contact support." (❌ Generic — it doesn't know your actual policy)
📄 RAG: [Searches company knowledge base] → Retrieved: "Section 4.2 — Digital Products: eligible for a full refund within 48 hours of purchase if the download link fails. No refund after 48 hours." → "Based on our policy: digital products can be refunded within 48 hours if the download fails." (✅ Accurate — grounded in real docs)
🤖 Agent [Thought]: This is a policy lookup. I'll search the knowledge base, check for any recent updates, and confirm the policy is current.
🤖 Agent [Actions]: search_docs("digital product refund policy") → search_docs("policy changes 2024") → check_date("policy last updated")
🤖 Agent: "Our current policy (updated March 2024): digital products qualify for a full refund within 48 hours if the download fails — recently extended from 24 hours." (✅ Accurate + current — but this was overkill. RAG was enough for this question.)

⚖️ Architecture Comparison at a Glance

💬
Chatbot

One LLM call. No retrieval, no tools, no loops. Fast and cheap. Knowledge limited to training data. High hallucination risk on specific facts.

Cost: Low · Speed: Fast · Reliability: High

📄
RAG

Retrieves relevant docs, then generates an answer grounded in them. Still single-shot — no loops, no tool use beyond retrieval. Reduces hallucination significantly.

Cost: Medium · Speed: Medium · Reliability: Medium-High

🤖
Agent

Reasons, calls tools, loops until the task is done. Handles complex multi-step tasks and can take real-world actions. More expensive and harder to predict.

Cost: High · Speed: Slow · Reliability: Medium-Low

🔀 Hybrid Approaches: The Real World

Production systems rarely use just one approach. The most common hybrid is Agentic RAG — an agent that uses document retrieval as one of its many tools. It can search the web, run calculations, and query a knowledge base — deciding for each task which combination of tools is needed.

Another common pattern is routing: an upfront classifier looks at the user's question and sends it to the cheapest system that can handle it — chatbot for simple questions, RAG for knowledge lookups, agent for complex multi-step tasks. This keeps costs reasonable while delivering full capability when it's actually needed.

🃏 RAG vs Agent — Know the Difference

📄 RAG System

Retrieves relevant documents from a knowledge base, then generates a response grounded in those documents. Single-shot by design — one retrieval step, one LLM call, one answer.

Strength: Accurate, grounded answers from your own data. Simple to build and maintain.

Limit: Can't take actions, can't iterate, no multi-step reasoning, no real-time tools.

🤖 Agent

Reasons about what to do, calls tools (which can include retrieval), observes results, and loops until the task is complete. Multi-step by design.

Strength: Handles complex tasks, takes real-world actions, adapts based on what it discovers.

Limit: More expensive, slower, less predictable. Don't use it when RAG is enough.

📖 Key Terms

RAG
Retrieval-Augmented Generation — searches a knowledge base for relevant documents and injects them into the LLM's prompt to ground the response in real data instead of training-data guesses.
Vector Database
A database that stores and searches text as numerical vectors (embeddings). Finds documents by meaning similarity, not exact keyword match. Powers the retrieval step in RAG.
Embedding
A numerical representation of text that captures its meaning. Similar texts get similar embeddings — which is how a vector database can find relevant documents even without matching exact words.
Grounding
Connecting an LLM's output to real, verifiable data to reduce hallucination. RAG is a grounding technique — the model answers from retrieved documents, not from what it "thinks" is true.
Agentic RAG
A hybrid pattern where an agent uses document retrieval as one tool among many. It can query your knowledge base AND search the web AND run code — whatever the task requires.
Routing
Directing incoming queries to the simplest system that can handle them — chatbot for simple Q&A, RAG for knowledge lookups, agent for complex multi-step tasks. Keeps costs low without sacrificing capability.
